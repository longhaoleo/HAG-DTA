README 里写的是：

```bash
python training_davis_kiba.py 0 1
```


其中：

- `dataset_id`: 0=davis, 1=kiba
- `model_id`: 0=GIN, 1=GCN, 2=GAT, 3=SAGE

### 4. Human 和 C.elegans 数据集划分与论文不一致

**状态：已修复（2026-04-29）**

论文写的是 8:1:1。

原代码逻辑是：

先划分 20% test，剩余 80% 再划分 20% validation。

实际比例是：训练集 64%，验证集 16%，测试集 20%。

**修复方式：**

`create_data_Human_Celegans.py` 中修改为真正的 8:1:1：

```python
train_data, test_data = train_test_split(df, test_size=0.1, stratify=df[2], random_state=42)
train_data, val_data = train_test_split(train_data, test_size=1/9, stratify=train_data[2], random_state=42)
```

先划 10% test，再从剩余 90% 中划出 1/9 作为 validation。这样最终就是 80%、10%、10%。

**注意：** 修改划分后需要重新运行 `create_data_Human_Celegans.py` 生成 .pt 文件，然后重新训练。

### 5. 回归任务存在用测试集选最优模型的风险（数据泄漏）

**状态：待修复**

`training_davis_kiba.py` 每个 epoch 都在 test_loader 上预测，并用 test MSE 保存最优模型。这会造成测试集参与模型选择，属于典型的数据泄漏风险。

返修建议：

1. 从训练数据中划出 validation set。
2. 每个 epoch 根据 validation MSE 保存最优模型。
3. 训练结束后只在 test set 上评估一次。
4. 论文中明确写清楚：validation 用于模型选择，test 只用于最终报告。

这是这篇论文返修中最重要的代码层面修复点之一。

### 6. Human 和 C.elegans 的 AUPRC 计算有隐藏错误

**状态：已修复（2026-04-29）**

`training_Human_Celegans.py` 中，验证集上计算了：

```python
tpr, fpr, _ = precision_recall_curve(G, P)
```

（注：原变量命名有误，`precision_recall_curve` 返回的是 precision、recall、thresholds，但被命名为 `tpr` 和 `fpr`。）

但当验证集结果变好后，代码切换到测试集预测，却没有重新计算测试集的 precision recall curve，导致 test AUPRC 可能沿用了验证集的曲线变量。

**修复方式：**

在测试集预测之后重新计算：

```python
G_test, P_test, pred_test = predicting(model, device, test_loader)
precision_test, recall_test, _ = precision_recall_curve(G_test, P_test)
auprc_test = auc(recall_test, precision_test)
best_ret = [roc_auc_score(G_test, P_test), auprc_test,
            precision_score(G_test, pred_test), recall_score(G_test, pred_test)]
```

同时修正了变量命名混淆的问题（原代码把 precision/recall 变量名叫 tpr/fpr）。

这个问题如果不修，分类任务结果的 AUPRC 会不可靠。
