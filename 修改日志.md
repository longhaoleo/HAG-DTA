### 4. 四个数据集的划分口径统一为 64% / 16% / 20%

**状态：已统一（2026-04-30）**

论文原先写成了 8:1:1，但当前项目准备统一回到一套更符合现有实现和复现实验的划分方案：

- train = 64%
- validation = 16%
- test = 20%

#### Human 和 C.elegans

这两个数据集的旧版实现本来就是：

```python
train_data, test_data = train_test_split(df, test_size=0.2, stratify=df[2], random_state=42)
train_data, val_data = train_test_split(train_data, test_size=0.2, stratify=train_data[2], random_state=42)
```

当前 Human / C.elegans 也统一使用：

1. 固定 `20%` 作为外层 `test`
2. 对剩余 `80%` 做 `5-fold stratified split`
3. 单次训练有效比例约为 `64% / 16% / 20%`

#### Davis 和 KIBA

原始 `create_data_davis_kiba.py` 直接把 5 个 training folds 全部展平合并，只保留了一个外层 test fold，因此没有单独保留 validation set。随后训练脚本又直接在测试集上做模型选择，这会导致验证阶段实际上使用了测试集信息。

当前统一方案是：

1. 先固定 `20%` 数据作为 `test`
2. 再对剩余 `80%` 做 `5-fold`
3. 每次取其中 1 折做 `validation`，其余 4 折做 `train`

因此单次训练的有效比例约为：

- train = 64%
- validation = 16%
- test = 20%

#### 论文表述

后续论文中的数据划分描述也需要统一修改为：

- all datasets used an outer test split of 20\%, and the remaining 80\% was further divided by five-fold cross-validation for model selection
- for each run, the effective train/validation/test ratio was approximately 64\% / 16\% / 20\%

**注意：** 修改划分后需要重新运行 `create_data_davis_kiba.py` 和 `create_data_Human_Celegans.py` 生成新的 `.pt` 文件，并重新训练。

### 5. 回归任务原先存在测试集参与模型选择的问题

**状态：已修正并纳入口径统一（2026-04-30）**

`training_davis_kiba.py` 最早的实现中，每个 epoch 都直接在 `test_loader` 上评估，并用 test MSE 保存最优模型。这会造成测试集参与模型选择，属于典型的数据泄漏风险。

同时，Davis / KIBA 的原始预处理并没有显式生成 validation set，导致“验证集”在实验实践中实际上被测试集替代使用。

现在统一处理为：

1. 预处理阶段固定外层 `test`，并生成内层 `5-fold` 索引
2. 训练阶段仅根据 validation MSE 保存最优模型
3. 训练结束后只在 test set 上报告最终结果

后续论文表述也应明确：

- validation is used for model selection
- test is used only for final evaluation

### 6. 注意力分数 CSV 跨种子累积的 bug

**状态：已修复（2026-04-29）**

`training_davis_kiba.py` 中，`a_list = []` 定义在第 78 行 `for seed` 循环的外部，导致每个种子结束时 a_list 没有被清空。其结果是：

- seed=100 训练 1000 epoch，注意力分数 CSV 有 1000 行（正确）
- seed=1000 训练 1000 epoch，注意力分数 CSV 有 2000 行（累积了前 1000 行）
- seed=2000 训练 1000 epoch，注意力分数 CSV 有 3000 行（累积了前 2000 行）

虽然 `random.csv`（最终指标）不受影响，因为 `random.csv` 只在种子循环结束后写入一次，但注意力分数 CSV 中的后续种子数据包含了前期种子的冗余行。

**修复方式：** 将 `a_list = []` 移到种子循环内部 `for seed in [100,1000,2000]:` 下面，与 `loss_train_list`、`loss_test_list` 并列。同时删除了种子循环外部的 `a_list = []`。

```python
for seed in [100, 1000, 2000]:
    same_seeds(seed)
    loss_train_list = []
    loss_test_list = []
    a_list = []  # 每轮种子循环重置
```

**注意：** 此前跑过的实验的注意力分数 CSV 文件（如 `davis注意力分数_1000_Diff_DTA_GIN.csv` 有 2000 行、`davis注意力分数_2000_Diff_DTA_GIN.csv` 有 3000 行）需要删除并重新训练以获取正确数据。`random.csv` 无需重跑。

### 7. Human 和 C.elegans 的 AUPRC 计算有隐藏错误

**状态：已修复（2026-04-29）**

`training_Human_Celegans.py` 中，验证集上计算了：

```python
tpr, fpr, _ = precision_recall_curve(G, P)
```

（注：原变量命名有误，`precision_recall_curve` 返回的是 precision、recall、thresholds，但被命名为 `tpr` 和 `fpr`。）

但当验证集结果变好后，代码切换到测试集预测，却没有重新计算测试集的 precision recall curve，导致 test AUPRC 可能沿用了验证集的曲线变量。

**修复方式：**

在测试集预测之后重新计算：

```python
G_test, P_test, pred_test = predicting(model, device, test_loader)
precision_test, recall_test, _ = precision_recall_curve(G_test, P_test)
auprc_test = auc(recall_test, precision_test)
best_ret = [roc_auc_score(G_test, P_test), auprc_test,
            precision_score(G_test, pred_test), recall_score(G_test, pred_test)]
```

同时修正了变量命名混淆的问题（原代码把 precision/recall 变量名叫 tpr/fpr）。

这个问题如果不修，分类任务结果的 AUPRC 会不可靠。

### 8. 训练输出目录切换到数据盘，避免系统盘空间不足

**状态：已修复（2026-04-30）**

原始训练脚本默认将模型文件和结果 CSV 直接保存在当前工作目录下。这在服务器环境中有两个问题：

1. 默认写入系统盘，长时间训练容易占满空间
2. 不同实验结果混在代码目录里，后续管理不方便

因此，`training_davis_kiba.py` 和 `training_Human_Celegans.py` 中新增了统一输出根目录参数：

```python
OUTPUT_ROOT = '/root/autodl-tmp/HAG-DTA-runs'
os.makedirs(OUTPUT_ROOT, exist_ok=True)
```

并统一通过：

```python
os.path.join(OUTPUT_ROOT, filename)
```

来保存以下产物：

- best checkpoint
- 训练/验证损失 CSV
- 注意力分数 CSV
- 随机种子结果汇总 CSV

#### 补充说明

`training_Human_Celegans.py` 中保存模型的文件名也同步补上了 `seed`，避免不同随机种子之间互相覆盖。

后续如果服务器目录需要调整，只需修改 `OUTPUT_ROOT` 一处即可。

### 9. 四个数据集统一切换到低占用 k-fold 方案

**状态：已修复（2026-04-30）**

为了同时满足这两个要求：

1. 四个数据集都使用 `validation` 做模型选择
2. 服务器磁盘不要为每个 fold 重复保存整套 `.pt`

当前四个数据集都已切换为：

- 只生成一份全量缓存 `*_all.pt`
- 每个 fold 只保存索引文件 `data/fold_indices/{dataset}_fold{0..4}.json`
- 训练脚本按 `fold_id` 从全量缓存中切出 `train / validation / test`

#### 当前数据组织

- `Davis`：`data/processed/davis_all.pt`
- `KIBA`：`data/processed/kiba_all.pt` 和 `data/processed/kiba_all1.pt`
- `Human`：`data/processed/Human_all.pt`
- `Celegans`：`data/processed/Celegans_all.pt`

以及：

- `data/fold_indices/davis_fold0.json` 到 `davis_fold4.json`
- `data/fold_indices/kiba_fold0.json` 到 `kiba_fold4.json`
- `data/fold_indices/Human_fold0.json` 到 `Human_fold4.json`
- `data/fold_indices/Celegans_fold0.json` 到 `Celegans_fold4.json`

#### 改动效果

- 不再需要为每个 fold 额外生成 5 套 `train/val/test.pt`
- 磁盘占用显著低于“每折一套 `.pt`”的做法
- 四个数据集统一使用：
  - 外层 `20% test`
  - 内层 `5-fold validation`
  - 单次训练有效比例约为 `64% / 16% / 20%`

#### 训练命令变化

两个训练脚本都新增了 `fold_id` 参数：

```bash

```

#### 论文表述建议

### 10. 新增集中路径配置，避免路径常量散落在脚本里

**状态：已修复（2026-05-01）**

原先几个主脚本中都直接写死了：

- `data/`
- `CACHE_ROOT`
- `data/processed/`

- `/root/autodl-tmp/HAG-DTA-runs`

这会带来两个问题：

1. 服务器切换目录时，要同时改多个脚本
2. 模型权重、预处理数据、fold 索引的位置不集中，后续容易改漏

现在新增了集中配置文件：

- [paths.py](/Users/leo/HAG-DTA/code_leo/config/paths.py:1)

- `DATA_ROOT`
- `CACHE_ROOT`
- `PROCESSED_DIR`
- `FOLD_DIR`
- `OUTPUT_ROOT`
- `CHECKPOINT_DIR`

#### 当前行为

- 预处理脚本和主训练脚本都会从 `config/paths.py` 读取路径
- 大体积的 `.pt` 缓存和 fold 索引默认放到 `/root/autodl-tmp/HAG-DTA-cache`
- 模型权重默认保存到 `CHECKPOINT_DIR`
- 结果 CSV 默认保存到 `OUTPUT_ROOT`
- 运行时会自动创建需要的目录

#### 可选覆盖方式

除了直接改 `paths.py`，现在也可以通过环境变量覆盖：

```bash
export HAG_DTA_DATA_ROOT=/path/to/data
export HAG_DTA_CACHE_ROOT=/root/autodl-tmp/HAG-DTA-cache
export HAG_DTA_OUTPUT_ROOT=/root/autodl-tmp/HAG-DTA-runs
```

这样就不需要再进每个训练脚本手动改路径常量。
