### 4. 四个数据集的划分口径统一为 64% / 16% / 20%

**状态：已统一（2026-04-30）**

论文原先写成了 8:1:1，但当前项目准备统一回到一套更符合现有实现和复现实验的划分方案：

- train = 64%
- validation = 16%
- test = 20%

#### Human 和 C.elegans

这两个数据集的旧版实现本来就是：

```python
train_data, test_data = train_test_split(df, test_size=0.2, stratify=df[2], random_state=42)
train_data, val_data = train_test_split(train_data, test_size=0.2, stratify=train_data[2], random_state=42)
```

因此 Human / C.elegans 现在已改回旧版 `64% / 16% / 20%`。

#### Davis 和 KIBA

原始 `create_data_davis_kiba.py` 直接把 5 个 training folds 全部展平合并，只保留了一个外层 test fold，因此没有单独保留 validation set。随后训练脚本又直接在测试集上做模型选择，这会导致验证阶段实际上使用了测试集信息。

为统一四个数据集的实验流程，Davis / KIBA 现改为显式生成：

- train = 64%
- validation = 16%
- test = 20%

对应逻辑为：

```python
train_idx, test_idx = train_test_split(pair_indices, test_size=0.2, random_state=42)
train_idx, val_idx = train_test_split(train_idx, test_size=0.2, random_state=42)
```

#### 论文表述

后续论文中的数据划分描述也需要统一修改为：

- all datasets were split into training, validation, and test sets with a ratio of 64\% / 16\% / 20\%

**注意：** 修改划分后需要重新运行 `create_data_davis_kiba.py` 和 `create_data_Human_Celegans.py` 生成新的 `.pt` 文件，并重新训练。

### 5. 回归任务原先存在测试集参与模型选择的问题

**状态：已修正并纳入口径统一（2026-04-30）**

`training_davis_kiba.py` 最早的实现中，每个 epoch 都直接在 `test_loader` 上评估，并用 test MSE 保存最优模型。这会造成测试集参与模型选择，属于典型的数据泄漏风险。

同时，Davis / KIBA 的原始预处理并没有显式生成 validation set，导致“验证集”在实验实践中实际上被测试集替代使用。

现在统一处理为：

1. 预处理阶段显式划分 train / validation / test = `64% / 16% / 20%`
2. 训练阶段仅根据 validation MSE 保存最优模型
3. 训练结束后只在 test set 上报告最终结果

后续论文表述也应明确：

- validation is used for model selection
- test is used only for final evaluation

### 6. 注意力分数 CSV 跨种子累积的 bug

**状态：已修复（2026-04-29）**

`training_davis_kiba.py` 中，`a_list = []` 定义在第 78 行 `for seed` 循环的外部，导致每个种子结束时 a_list 没有被清空。其结果是：

- seed=100 训练 1000 epoch，注意力分数 CSV 有 1000 行（正确）
- seed=1000 训练 1000 epoch，注意力分数 CSV 有 2000 行（累积了前 1000 行）
- seed=2000 训练 1000 epoch，注意力分数 CSV 有 3000 行（累积了前 2000 行）

虽然 `random.csv`（最终指标）不受影响，因为 `random.csv` 只在种子循环结束后写入一次，但注意力分数 CSV 中的后续种子数据包含了前期种子的冗余行。

**修复方式：** 将 `a_list = []` 移到种子循环内部 `for seed in [100,1000,2000]:` 下面，与 `loss_train_list`、`loss_test_list` 并列。同时删除了种子循环外部的 `a_list = []`。

```python
for seed in [100, 1000, 2000]:
    same_seeds(seed)
    loss_train_list = []
    loss_test_list = []
    a_list = []  # 每轮种子循环重置
```

**注意：** 此前跑过的实验的注意力分数 CSV 文件（如 `davis注意力分数_1000_Diff_DTA_GIN.csv` 有 2000 行、`davis注意力分数_2000_Diff_DTA_GIN.csv` 有 3000 行）需要删除并重新训练以获取正确数据。`random.csv` 无需重跑。

### 7. Human 和 C.elegans 的 AUPRC 计算有隐藏错误

**状态：已修复（2026-04-29）**

`training_Human_Celegans.py` 中，验证集上计算了：

```python
tpr, fpr, _ = precision_recall_curve(G, P)
```

（注：原变量命名有误，`precision_recall_curve` 返回的是 precision、recall、thresholds，但被命名为 `tpr` 和 `fpr`。）

但当验证集结果变好后，代码切换到测试集预测，却没有重新计算测试集的 precision recall curve，导致 test AUPRC 可能沿用了验证集的曲线变量。

**修复方式：**

在测试集预测之后重新计算：

```python
G_test, P_test, pred_test = predicting(model, device, test_loader)
precision_test, recall_test, _ = precision_recall_curve(G_test, P_test)
auprc_test = auc(recall_test, precision_test)
best_ret = [roc_auc_score(G_test, P_test), auprc_test,
            precision_score(G_test, pred_test), recall_score(G_test, pred_test)]
```

同时修正了变量命名混淆的问题（原代码把 precision/recall 变量名叫 tpr/fpr）。

这个问题如果不修，分类任务结果的 AUPRC 会不可靠。
