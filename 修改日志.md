### 6. 注意力分数 CSV 跨种子累积的 bug

**状态：已修复（2026-04-29）**

`training_davis_kiba.py` 中，`a_list = []` 定义在第 78 行 `for seed` 循环的外部，导致每个种子结束时 a_list 没有被清空。其结果是：

- seed=100 训练 1000 epoch，注意力分数 CSV 有 1000 行（正确）
- seed=1000 训练 1000 epoch，注意力分数 CSV 有 2000 行（累积了前 1000 行）
- seed=2000 训练 1000 epoch，注意力分数 CSV 有 3000 行（累积了前 2000 行）

虽然 `random.csv`（最终指标）不受影响，因为 `random.csv` 只在种子循环结束后写入一次，但注意力分数 CSV 中的后续种子数据包含了前期种子的冗余行。

**修复方式：** 将 `a_list = []` 移到种子循环内部 `for seed in [100,1000,2000]:` 下面，与 `loss_train_list`、`loss_test_list` 并列。同时删除了种子循环外部的 `a_list = []`。

```python
for seed in [100, 1000, 2000]:
    same_seeds(seed)
    loss_train_list = []
    loss_test_list = []
    a_list = []  # 每轮种子循环重置
```

**注意：** 此前跑过的实验的注意力分数 CSV 文件（如 `davis注意力分数_1000_Diff_DTA_GIN.csv` 有 2000 行、`davis注意力分数_2000_Diff_DTA_GIN.csv` 有 3000 行）需要删除并重新训练以获取正确数据。`random.csv` 无需重跑。

### 7. Human 和 C.elegans 的 AUPRC 计算有隐藏错误

**状态：已修复（2026-04-29）**

`training_Human_Celegans.py` 中，验证集上计算了：

```python
tpr, fpr, _ = precision_recall_curve(G, P)
```

（注：原变量命名有误，`precision_recall_curve` 返回的是 precision、recall、thresholds，但被命名为 `tpr` 和 `fpr`。）

但当验证集结果变好后，代码切换到测试集预测，却没有重新计算测试集的 precision recall curve，导致 test AUPRC 可能沿用了验证集的曲线变量。

**修复方式：**

在测试集预测之后重新计算：

```python
G_test, P_test, pred_test = predicting(model, device, test_loader)
precision_test, recall_test, _ = precision_recall_curve(G_test, P_test)
auprc_test = auc(recall_test, precision_test)
best_ret = [roc_auc_score(G_test, P_test), auprc_test,
            precision_score(G_test, pred_test), recall_score(G_test, pred_test)]
```

同时修正了变量命名混淆的问题（原代码把 precision/recall 变量名叫 tpr/fpr）。

这个问题如果不修，分类任务结果的 AUPRC 会不可靠。

### 8. 训练输出目录切换到数据盘，避免系统盘空间不足

**状态：已修复（2026-04-30）**

原始训练脚本默认将模型文件和结果 CSV 直接保存在当前工作目录下。这在服务器环境中有两个问题：

1. 默认写入系统盘，长时间训练容易占满空间
2. 不同实验结果混在代码目录里，后续管理不方便

因此，`training_davis_kiba.py` 和 `training_Human_Celegans.py` 中新增了统一输出根目录参数：

```python
OUTPUT_ROOT = '/root/autodl-tmp/HAG-DTA-runs'
os.makedirs(OUTPUT_ROOT, exist_ok=True)
```

并统一通过：

```python
os.path.join(OUTPUT_ROOT, filename)
```

来保存以下产物：

- best checkpoint
- 训练/验证损失 CSV
- 注意力分数 CSV
- 随机种子结果汇总 CSV

#### 补充说明

`training_Human_Celegans.py` 中保存模型的文件名也同步补上了 `seed`，避免不同随机种子之间互相覆盖。

后续如果服务器目录需要调整，只需修改 `OUTPUT_ROOT` 一处即可。

### 11. 废弃 _all.pt + Subset，切换为 per-fold 预分割 .pt 文件

**状态：已修复（2026-05-02）**

第 9 条的 `_all.pt + Subset + DenseDataLoader` 方案导致训练速度从 18s/epoch 降至 500s/epoch（27× 变慢）。根因是已 deprecated 的 `DenseDataLoader` 在包装 `torch.utils.data.Subset` 时存在严重性能退化。

**替代方案：**
- 数据预处理阶段直接为每个 fold 生成独立的 `train/val/test .pt` 文件
- 训练脚本直接加载预分割 `.pt`，移除 Subset / build_position_map / subset_from_original_ids
- 存储增加约 5×（每 fold 复制 3 份），训练速度恢复 18s/epoch
- JSON fold 索引保留仅作参考

**涉及文件：**
- `create_data_davis_kiba.py` — 新增 `save_fold_pt()`，循环预生成 `davis_fold{N}_{train,val,test}.pt`
- `create_data_Human_Celegans.py` — 同上
- `training_davis_kiba.py` — 直接加载 `{dataset}_fold{fold_id}_train.pt` 等
- `training_Human_Celegans.py` — 同上

**命名规则：**

```
{CACHE_ROOT}/processed/{dataset}_fold{fold_id}_{train,val,test}.pt
```

例如：`davis_fold0_train.pt`, `davis_fold3_val.pt`, `kiba_sub1_fold4_test.pt`

### 16. TransformerCPI 显著性比较采用“同样本集公平比较”原则

**状态：已记录（2026-05-09）**

官方 `TransformerCPI` 的 `mol_featurizer.py` 会过滤掉带 `'.'` 的 SMILES。

但当前原始分类数据中，这类样本数量不可忽略：
- Human: `516`
- C.elegans: `275`

如果直接沿用官方过滤逻辑，会导致 TransformerCPI 实际训练样本集与 HAG-DTA 不一致，从而影响后续显著性比较的公平性。

**因此本轮返修统一决定：**
- TransformerCPI 仍参考官方模型结构与训练风格
- 但比较时使用与 HAG-DTA 完全相同的 Human / C.elegans 原始 txt 数据
- 使用相同的 `80/10/10` 划分
- 不采用官方“过滤带 `.` 的 SMILES”这一步

**结论：**
当前集成到 `code_base/` 的 TransformerCPI 是“官方结构对齐 + 比较样本集对齐”的返修比较版本。
